# Test Data Management — Seed/Reset стратегія (UA)

## Ціль
Зробити E2E тести швидкими, повторюваними і не флейковими через **передбачуваний стан апки** на старті кожного тесту.

У навчальних репозиторіях найбільше часу з'їдає створення даних через UI знову і знову.

---

## Рекомендований підхід (QA-friendly)
**In-memory репозиторій + reset hook + debug-only дії**.

### Чому саме так
- Працює офлайн
- Не потрібні зовнішні сервіси
- Детерміновано і швидко
- Вчить best practices тестованості

---

## Пропозиція дизайну для training app

### 1) Режими виконання
- `debug` / `profile`: дозволяємо seed/reset
- `release`: seed/reset вимкнені

### 2) Точки входу для Seed/Reset (обери одну)

#### Варіант A — Debug menu в UI (найкраще для навчання)
Прихований екран "Debug" доступний тільки в debug збірці:
- Reset state
- Seed demo notes (N=3)
- Login як test user

Плюси: зрозуміло і наочно.
Мінуси: все одно UI-кроки.

#### Варіант B — Deep link / route command (краще для автоматизації)
Debug-only route:
- `/debug/reset`
- `/debug/seed?notes=3`

E2E тест може переходити напряму.

#### Варіант C — Platform channel команда
Метод-канал для reset/seed.
Плюси: потужно; Мінуси: складніше.

---

## Мінімальні критерії (для Week 2 E2E)
- Є 1 команда, яка скидає апку в відомий стан.
- Є 1 команда, яка створює seed-дані (наприклад 3 notes).
- Команди доступні тільки в debug/profile.

---

## Що тестуємо
- Unit: логіка seed/reset
- Widget: debug screen існує тільки в debug
- E2E: перший крок завжди `reset + seed`
